# `contracts/` — Cross-Language Wire Contracts

This directory is the **single source of truth** for every message
that crosses a domain boundary in DIX VISION v42.2. It implements
Core Guarantee #4 from `docs/DIX_VISION_v42_2_COMPILED.md`:

> Typed communication only.

Once a module is ported to Rust or Go, it MUST communicate with the
rest of the system through the Protobuf messages defined here — no
ad-hoc dicts, no JSON-over-the-wire, no string-keyed payloads.

---

## Files

| File | Owns |
|---|---|
| `market.proto` | MARKET-domain messages: ticks, quotes, frames. Produced by Indira's market graph. |
| `execution.proto` | Order intents, order events, fills, adapter state. Produced by `mind`; consumed by `execution/`. |
| `governance.proto` | Policy decisions, approval requests, risk constraints, governance transitions. |
| `system.proto` | Cross-domain system signals: `SYSTEM_HAZARD`, kill-switch events, config-change events, feature-flag deltas. |
| `ledger.proto` | Append-only ledger events + snapshot format. Consumed by `state/ledger/` (Go) and the replay tools. |

---

## Invariants (enforced by `tools/contract_diff.py`)

1. **Field numbers never change.** Adding a new field picks the next
   unused number. Removing a field reserves its number forever. This
   is how backward compatibility (Tier-0 Step 15) is mechanically
   enforced on every PR.
2. **No field ever changes type.** If a field's semantics changes,
   it gets a new number. The old field is reserved.
3. **Every cross-domain enum must be fully enumerated.** `UNKNOWN = 0`
   is mandatory for forward compatibility.
4. **No `google.protobuf.Any` in cross-domain messages.** Any field
   that would need it must be resolved to a concrete sub-message
   instead. This keeps the wire self-describing across languages.
5. **Package names are stable.** `dixvision.v42_2.<domain>` (Protobuf
   package identifiers cannot contain `.` between digits without
   being parsed as a package boundary, so `v42_2` uses an underscore).

---

## Generation

Python stubs are generated by `tools/gen_protos.sh` into
`contracts/gen/python/`. Rust stubs (`prost-build`) and Go stubs
(`protoc-gen-go`) land in subsequent PRs that introduce those
toolchains.

Generated files are **not** committed. CI regenerates them on every
build and fails if the checked-in `.proto` files don't parse.

---

## Adding a new message

1. Pick a domain. If the message crosses two domains, it belongs in
   whichever domain **produces** it.
2. Assign the next unused field number.
3. Add a doc comment above the message explaining the invariant it
   preserves (e.g. "emitted exactly once per order intent").
4. Run `tools/gen_protos.sh` locally and verify the Python stubs
   import cleanly.
5. Open a PR. `contract_diff.py` will fail the build if the change
   violates the invariants above.
